Por “kata”, se entende um sistema que foi desenvolvido com o objetivo de contribuir para o desenvolvimento da comunidade de desenvolvedores e que contém atividades de refatoração e testes, por exemplo, como o Gilded Rose Kata, disponibilizado por Emily Bache. Projetos como esse propõem exercícios de programação focados em prática técnica. O Gilded Rose, por exemplo, apresenta estruturas típicas de código legado, como o if “aninhado” desenvolvido pelo antigo funcionário fictício, Leeroy e um teste inicial propositalmente quebrado, que serve de ponto de partida para o desenvolvedor. Como muitos desenvolvedores de software afirmam, más práticas e códigos desenvolvidos como este são situações encontradas com frequência.

De acordo com Emily Bache, em seu vídeo no YouTube “Why Developers LOVE The Gilded Rose Kata”, a linguagem de programação utilizada pelo criador do projeto original foi C#, a qual escolhi para a refatoração do mesmo por ter maior afinidade e contato cotidiano para estudos e desenvolvimento de projetos pessoais back-end.

O primeiro passo foi compreender o código legado existente no método UpdateQuality da classe GildedRose. O código original utilizava estruturas if/else profundamente aninhadas que verificavam o nome do item repetidamente ao longo do método, tornando a lógica difícil de compreender, algo que torna, por sua vez, a manutenção dificultosa.

Antes de qualquer alteração no código de produção, escrevi testes automatizados com xUnit cobrindo todos os comportamentos descritos nos requisitos: item normal antes e após a data de venda, Aged Brie, Sulfuras e Backstage Pass – incluindo casos de borda como quality em 0, quality em 50 e SellIn negativo. Essa etapa foi essencial para garantir que a refatoração não alteraria o comportamento existente.

Com os testes estabelecidos, refatorei o método UpdateQuality. A solução adotada foi extrair a lógica de cada tipo de item para métodos privados dedicados (UpdateNormalItem, UpdateAgedBrie, UpdateBackstagePass, UpdateConjured), coordenados por um método UpdateItem com um switch sobre o nome do item. Centralizei o decremento de SellIn seguindo o princípio DRY, já que ele é comum a todos os itens exceto Sulfuras.

Com a refatoração concluída e todos os testes efetuados com sucesso, o item Conjured – que degrada o dobro da velocidade de um item normal – foi desenvolvido com a inclusão de um único método e um case no switch.

Pessoalmente, o projeto foi desafiador em algumas etapas. Inicialmente, despendi um bom tempo analisando os ifs “aninhados” e como eles se relacionavam com a classe “Program”. A lógica do projeto parece simples à primeira vista, mas ao aplicar os testes unitários, ela se torna mais clara e, por vezes, complexa porque a aplicação de testes automatizados exige cautela e análise clara do que ocorre tanto nas regras de negócio, neste caso, disponíveis no README do projeto, quanto do que acontece no projeto em si. 

Precisei gastar bastante tempo resolvendo um bug simples que eu mesmo criei durante o desenvolvimento dos testes, porém fiquei bastante satisfeito por ter conseguido desenvolvê-los com relativa facilidade. Embora a sintaxe dos testes mude de uma linguagem para outra, o desenvolvimento não foi difícil pois o conceito é o mesmo. Algo que mereceu a minha atenção antes de começar a refatoração foi a relação com os direitos autorais do repositório original e a licença do MIT, os quais mantive preservados na raiz do projeto.

Ademais, durante o desenvolvimento do projeto ficou claro para mim a importância da realização de testes durante o processo de refatoração de software, já que, mesmo que o software desenvolvido careça de boas práticas de engenharia ele pode funcionar. Neste caso, para garantir que o que já funciona continue funcionando, utilizar testes automatizados é uma boa proposta de modelo de desenvolvimento a ser utilizado.
